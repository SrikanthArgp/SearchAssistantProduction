### Decision (2026-07-03) — New Phase 7: Next.js frontend, inserted (real renumbering, not a late-numbered bookkeeping slot)
User wants a Next.js frontend so this becomes an actual fullstack app instead of an API with no client. Clarified three things before touching `plan.md`: feature depth, token-storage strategy, and phase placement. Docs-only pass.
- **Feature depth: full chat-app UX**, not a bare MVP — session rename/delete/archive, streaming render, loading/error states. More build time than a minimal API-smoke-test UI, but matches the user's stated "build the product end-to-end" learning goal.
- **Token storage: access token in memory (React context) + direct Bearer-header fetches to FastAPI**, no BFF/httpOnly-cookie proxy layer. Simpler than the BFF alternative discussed, at the cost of the token living in browser JS memory (XSS-exposure tradeoff, same category of decision as the Phase 3 bcrypt/passlib call).
  - Consequence not initially spelled out by the user, surfaced during planning: access-token-only-in-memory means a page reload would force re-login unless the *refresh* token persists somewhere. Resolved as: refresh token in `localStorage` (accepted XSS-exposure tradeoff on the refresh token specifically, mitigated by the already-built short access-token TTL + Redis revocation check from Phase 3), proactive silent refresh at ~80% of the 15-minute access-token TTL.
  - Second consequence surfaced during planning: native `EventSource` (the obvious choice for consuming the SSE stream endpoint) cannot send custom headers, so it can't carry the `Authorization: Bearer` token this auth design requires. Resolved as: `fetch` + manual `ReadableStream` parsing instead of `EventSource`.
- **Phase placement — real renumbering, not the "numbered last" trick used for Langfuse/OTel**: user explicitly asked for this as an actual **Phase 7**, right after Phase 6 (FastAPI) and before Phase 8 (Evaluation, formerly Phase 7). My initial suggestion was to number it last (Phase 13) but build it early, mirroring the Langfuse precedent — user rejected that in favor of true insertion. Rationale discussed: Phases 8–13 (eval/tests/hardening/observability/CI) don't change the API contract, so nothing downstream is broken by inserting here; building the UI this early surfaces real contract problems (SSE shape, pagination, CORS, error envelope) while still cheap to fix. Any non-trailing base64url character always encodes a full 6 real bits (only the very last character of a segment can land on unused padding bits), and changing a payload byte always changes what the signature was computed over — so this is deterministic, not a probabilistic improvement. Protection is a client-side check in `app/chat/layout.tsx`, redirecting to `/login` once `AuthProvider`'s status resolves to `unauthenticated`.

### Retrofit (2026-07-04) — Two real backend bugs found while smoke-testing Phase 7, fixed same session
Neither is Phase 7 code — both are in already-shipped Phases 1–6 — but both were surfaced by actually driving the new frontend against the real backend with Redis down (Docker Desktop wasn't running this session), and the user asked to fix both immediately rather than defer.

- **Redis client had no connect/socket timeout, so "fail open" was true but slow.** Every Redis-touching path (Phase 3's revocation check, Phase 4's cache, Phase 6's auth rate limiter) already degrades correctly on `RedisError` — but `redis.asyncio.from_url(...)` was called with no `socket_connect_timeout`/`socket_timeout` in either `api/main.py` (the real `app.state.redis`, Phase 6) or `auth/dependencies.py`'s `get_redis_client()` (the default provider Phase 6 overrides via `app.dependency_overrides`, kept in sync for the rare path where it isn't). Without an explicit timeout, redis-py falls back to the OS's own TCP connect timeout — measured at ~5–6 seconds per Redis call while Docker was down, meaning a single `/auth/logout` (two `_revoke_in_cache` calls) took **12.6 seconds** end to end. Fixed by adding `socket_connect_timeout=2, socket_timeout=2` to both call sites. Re-measured after the fix: register dropped from ~4.95s to ~3.4s (bcrypt + one still-attempted-but-now-fast Redis timeout + real Supabase round trip); this is entirely a Redis-down-in-this-session artifact — with Redis actually reachable, none of these paths ever hit the timeout at all.
- **Intermittent 500 on `/auth/login` and `/auth/register`**: `sqlalchemy.exc.InvalidRequestError: Could not refresh instance` in `db/crud/refresh_tokens.py`'s `create()`, at `await db.refresh(row)` immediately after a successful `await db.flush()` — i.e. the row was inserted but the very next SELECT-by-PK couldn't see it. Reproduced with plain backend `httpx`/`curl` calls, no frontend involved, ruling out anything React-side. Root cause: `DATABASE_URL` points at **Supabase's Transaction pooler (pgbouncer, port 6543)**, which can hand a different backend Postgres process to the same client connection between transactions; psycopg3 auto-prepares statements after repeated use (`prepare_threshold=5` by default) and caches them per client connection, so a statement prepared against one backend process can get silently executed against a different one after pgbouncer swaps — this is a documented psycopg/Supabase incompatibility with pgbouncer transaction-mode pooling, not something specific to this codebase. Fixed with the standard documented mitigation: `connect_args={"prepare_threshold": None}` on `db/base.py`'s `create_async_engine(...)`, disabling statement preparation entirely. **Verified fixed, not just plausible**: 8 sequential logins and 10 concurrent logins against the same user both went 100% green after the fix (the sequential case alone had failed on the very first call before it, reproducibly).
- **Regression**: `pytest tests/ multi_agent/chains/tests/ -m "not integration"` → 93 passed, 11 failed — all 11 pre-existing (10 `requires_redis` tests, since Docker was down all session; 1 is the long-documented `test_hallucination_grader_answer_no` LLM-judgment flake). No new failures from either fix.
- No `plan.md` changes — both are bug fixes to already-correct designs (the fail-open behavior and the pgbouncer transaction-pooler choice were both deliberate, documented decisions; only the missing timeout and the missing `prepare_threshold` setting were bugs).
