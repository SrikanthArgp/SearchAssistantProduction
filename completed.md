### Retrofit (2026-07-04) ‚Äî Two real backend bugs found while smoke-testing Phase 7, fixed same session
Neither is Phase 7 code ‚Äî both are in already-shipped Phases 1‚Äì6 ‚Äî but both were surfaced by actually driving the new frontend against the real backend with Redis down (Docker Desktop wasn't running this session), and the user asked to fix both immediately rather than defer.

- **Redis client had no connect/socket timeout, so "fail open" was true but slow.** Every Redis-touching path (Phase 3's revocation check, Phase 4's cache, Phase 6's auth rate limiter) already degrades correctly on `RedisError` ‚Äî but `redis.asyncio.from_url(...)` was called with no `socket_connect_timeout`/`socket_timeout` in either `api/main.py` (the real `app.state.redis`, Phase 6) or `auth/dependencies.py`'s `get_redis_client()` (the default provider Phase 6 overrides via `app.dependency_overrides`, kept in sync for the rare path where it isn't). Without an explicit timeout, redis-py falls back to the OS's own TCP connect timeout ‚Äî measured at ~5‚Äì6 seconds per Redis call while Docker was down, meaning a single `/auth/logout` (two `_revoke_in_cache` calls) took **12.6 seconds** end to end. Fixed by adding `socket_connect_timeout=2, socket_timeout=2` to both call sites. Re-measured after the fix: register dropped from ~4.95s to ~3.4s (bcrypt + one still-attempted-but-now-fast Redis timeout + real Supabase round trip); this is entirely a Redis-down-in-this-session artifact ‚Äî with Redis actually reachable, none of these paths ever hit the timeout at all.
- **Intermittent 500 on `/auth/login` and `/auth/register`**: `sqlalchemy.exc.InvalidRequestError: Could not refresh instance` in `db/crud/refresh_tokens.py`'s `create()`, at `await db.refresh(row)` immediately after a successful `await db.flush()` ‚Äî i.e. the row was inserted but the very next SELECT-by-PK couldn't see it. Reproduced with plain backend `httpx`/`curl` calls, no frontend involved, ruling out anything React-side. Root cause: `DATABASE_URL` points at **Supabase's Transaction pooler (pgbouncer, port 6543)**, which can hand a different backend Postgres process to the same client connection between transactions; psycopg3 auto-prepares statements after repeated use (`prepare_threshold=5` by default) and caches them per client connection, so a statement prepared against one backend process can get silently executed against a different one after pgbouncer swaps ‚Äî this is a documented psycopg/Supabase incompatibility with pgbouncer transaction-mode pooling, not something specific to this codebase. Fixed with the standard documented mitigation: `connect_args={"prepare_threshold": None}` on `db/base.py`'s `create_async_engine(...)`, disabling statement preparation entirely. **Verified fixed, not just plausible**: 8 sequential logins and 10 concurrent logins against the same user both went 100% green after the fix (the sequential case alone had failed on the very first call before it, reproducibly).
- **Regression**: `pytest tests/ multi_agent/chains/tests/ -m "not integration"` ‚Üí 93 passed, 11 failed ‚Äî all 11 pre-existing (10 `requires_redis` tests, since Docker was down all session; 1 is the long-documented `test_hallucination_grader_answer_no` LLM-judgment flake). No new failures from either fix.
- No `plan.md` changes ‚Äî both are bug fixes to already-correct designs (the fail-open behavior and the pgbouncer transaction-pooler choice were both deliberate, documented decisions; only the missing timeout and the missing `prepare_threshold` setting were bugs).

### Phase 13 ‚Äî Observability (Langfuse)

- **Real bug found: `langfuse.propagate_attributes()` silently loses tagging under `graph.ainvoke()`.** The first implementation wrapped `_graph_config()`'s `ainvoke()` call in `with propagate_attributes(trace_name=..., user_id=..., session_id=...)`, per `plan.md`'s own step-2 wording ‚Äî this is also literally what Langfuse's official LangGraph cookbook recommends. Builds a `TracerProvider`, conditionally attaches a `BatchSpanProcessor(OTLPSpanExporter(...))` only when `OTEL_EXPORTER_OTLP_ENDPOINT` + `GRAFANA_OTLP_INSTANCE_ID` + `GRAFANA_OTLP_TOKEN` are all present in `config.py`'s `Settings` (all three added as optional fields), then instruments `FastAPIInstrumentor`/`SQLAlchemyInstrumentor`/`RedisInstrumentor` against that provider regardless. Outside of an active span (e.g. at import time, or in a test with no request in flight), the span context is invalid and the keys are simply omitted ‚Äî confirmed the existing Phase 12 logging tests (which assert exact key sets) still pass unchanged.

Fixed via a `.env` file in the runner's root (`C:\gh-runner\.env`, a documented self-hosted-runner feature) prepending Git's `bin` to `PATH`.
4. **LocalStack's `ecr get-authorization-token` returns an empty `authorizationData` list** (crashing `aws ecr get-login-password` with "list index out of range") until at least one repository exists in the account/region ‚Äî confirmed by reproducing directly, then confirming that creating one repo fixes it. Real chicken-and-egg: a plain `terraform apply` can't run first either, since `aws_lambda_function` needs the image to already exist. Fixed by splitting the infra-changed path into `terraform init` ‚Üí targeted `apply -target=aws_ecr_repository.backend` ‚Üí image build/push ‚Üí full `apply`, inside the workflow itself (the same two-pass shape Phase 15/16's local applies already used manually).
5. **`var.secrets` has no default and `secrets.auto.tfvars` is gitignored** ‚Äî absent from any fresh runner checkout, so `terraform apply` failed immediately (Terraform requires every declared variable resolved before processing *any* resource, even a `-target`'d one). Fixed for LocalStack testing only via a new `LOCALSTACK_SECRETS_JSON` GitHub encrypted Secret (same values as the local `secrets.auto.tfvars`, added with explicit user sign-off since real credentials were involved) wired to `TF_VAR_secrets`. **Real AWS deploys still have no answer to this ‚Äî open follow-up**, not solved here.
6. **MSYS/Git-Bash argv path-mangling** silently rewrote `aws ssm get-parameter --name /crag/.../cloudfront_domain` into a nonsense Windows path, producing a confusing `"can't be prefixed with ssm"` SSM validation error that had nothing to do with the actual parameter. Same class of bug `grand-enterprize-deploy-steps.md` already documents for frontend builds. Fixed with `MSYS_NO_PATHCONV: "1"` at the job level.
7. **`jq` isn't installed on this Windows runner** (ubuntu-latest ships it by default, masking the gap until actually running here) ‚Äî `cd-ecs.yml`'s task-definition-patch step failed with exit 127. Installed via `winget install jqlang.jq`, added to the runner's `.env` PATH override.

Two further gaps found once the above were clear and both `terraform apply`s actually succeeded ‚Äî both go deeper than the CD workflows themselves:

8. **`/health` 404'd through both CloudFront distributions.** `api/main.py` mounts `/health` deliberately outside the `/v1` prefix, but both `cloudfront.tf`s only route `/v1/*` to the backend ‚Äî everything else, including `/health`, fell through to the S3 frontend origin's `default_cache_behavior` (a `NoSuchKey` 404, not a real health-check failure). Every prior Phase 15/16 verification session tested `/health` directly against the origin (ALB/Function URL), never through CloudFront, so this was never caught before `cd-lambda.yml`/`cd-ecs.yml`'s smoke check tried it. **Real infra fix, both stacks, user-confirmed**: added a matching `/health` `ordered_cache_behavior` to `infra/lambda-gate/cloudfront.tf` and `infra/fargate/cloudfront.tf`.

ECS's `/health` didn't surface this (`SELECT 1` + Redis `PING` only, no graph invocation), but a real chat request there would hit the identical crash. **How the real CD pipeline should provision the vector store (bake ingestion into the image build, commit the DB, a separate provisioning step) is an open design question, deliberately not decided.** Per explicit user direction ("consider it pre-populated for simplicity"), `cd-lambda.yml`/`cd-ecs.yml` gained a LocalStack-only step that copies this dev machine's own populated `.chroma/` into the checkout before the Docker build ‚Äî a `localstack`-mode simplification that cannot exist on a GitHub-hosted runner, not a production answer.

**Follow-up (same session): frontend deploy added to both CD workflows.** Manually poking the live URLs after the above surfaced a tenth real gap ‚Äî frontend deploy was entirely absent from Phase 18/19's original scope (`cd-lambda-deploy-steps.md`/`cd-ecs-deploy-steps.md` never mention S3 or the frontend at all), even though Phase 15/16 provision a frontend bucket and a static-asset CloudFront behavior; `sync_frontend.sh` was the only way to populate it, always run by hand. **Follow-up (2026-07-12): re-running both CD workflows end to end against a genuinely fresh LocalStack instance (state wiped, not the one left running from the prior session) surfaced two more real gaps, both fixed.** First, confirmed the already-known chicken-and-egg (gap 4/9 above) recurs any time LocalStack state is wiped and the triggering commit doesn't touch `infra/lambda-gate/**`/`infra/fargate/**` ‚Äî both workflows' path-filters correctly take the fast path (assumes existing infra) and there's nothing there.
